# Core Workflow

## Step 1：理解用户意图，提取关键词和筛选参数

**1. 提取关键词**

- 从用户描述中提取 2~6 字的搜索关键词
- 若用户意图模糊（如"帮我搜B站账号"），主动询问：「请问你想搜索哪个领域或方向的B站账号？」
- 不得在用户未提供关键词时擅自猜测并调用脚本

**2. 识别筛选参数**（用户未指定时使用默认值）

- **排序方式**（`--order`）：固定 `fans`（按粉丝数排序），无需用户指定
  - 所有查询统一按粉丝数降序输出
- **页码**（`--page`）：默认 `1`
  - 用户回复「查看更多」→ 页码 +1
  - 未触发「查看更多」→ `1`（默认）

## Step 2：调用搜索脚本

```bash
python3 "$SKILL_PATH/scripts/search_bili_user.py" "<关键词>" [--order 排序] [--page 页码]
```

**参数说明：**

| 参数       | 可选值                  | 含义                                      |
| ---------- | ----------------------- | ----------------------------------------- |
| keyword    | 任意字符串              | 搜索关键词                                 |
| `--order`  | `totalrank` / `fans`    | 综合排序 / 按粉丝数排序（固定传 `fans`）      |
| `--page`   | 正整数                  | 页码，从1开始（默认1）                      |

脚本以 JSON 格式输出：

| 字段           | 说明                          |
| -------------- | ----------------------------- |
| `users`        | 账号列表                      |
| `order_label`  | 本次排序方式文字说明           |
| `page`         | 当前页码                      |
| `has_next`     | 是否有下一页（true/false）    |
| `total_count`  | 本页账号数量                  |

每个账号字段：

| 字段           | 说明                                      |
| -------------- | ----------------------------------------- |
| `uid`          | B站用户UID                               |
| `nickname`     | 昵称                                      |
| `avatar`       | 头像链接（https协议）                      |
| `description`  | 个人简介                                  |
| `fans_num`     | 粉丝数                                    |
| `gender`       | 性别 (1=男 2=女 3=未知)                   |
| `level`        | B站等级 (0-6)                             |
| `is_living`    | 是否正在直播                               |
| `is_uploader`  | 是否UP主                                  |
| `live_room_id` | 直播间ID（0表示无直播间）                  |
| `space_url`    | B站个人主页链接（直接可用）                |

## Step 3：智能分类并展示

### 情况 A：users 数量 > 0（有匹配结果）

**A1. AI 智能分类**

获取到账号列表后，AI 必须根据每个账号的 `nickname`（昵称）和 `description`（简介）自动判断其内容方向，将账号归入以下分类体系：

| 分类 | 典型方向 |
|------|----------|
| 🎮 游戏 | 游戏解说、游戏直播、手游官方、主机游戏、电竞 |
| 😄 生活娱乐/搞笑 | 搞笑段子、生活日常、精分、娱乐杂谈 |
| 🎬 影视/解说 | 影视剧解说、动漫、影评、剪辑 |
| 🍜 美食 | 美食探店、吃播、烹饪 |
| 🔬 科技/知识 | 硬核科技、科普、编程、机器人 |
| 🏀 运动/户外 | 篮球、健身、舞蹈、户外运动 |
| 🎨 手工/创意 | 手工DIY、绘画、创意设计 |
| 📍 本地/旅行 | 本地生活、旅行vlog、城市探索 |
| 🎵 音乐 | 翻唱、乐器、音乐制作 |
| 📚 教育/读书 | 教程、读书分享、知识分享 |
| 🛒 带货/种草 | 好物推荐、种草、购物分享 |
| 🏢 官方/机构 | 官方账号、企业号、媒体号 |

- 每个账号必须归入且仅归入一个最匹配的分类
- 若某分类仅含 1 个账号，仍单独成类，不合并
- AI 须为每个账号生成 2~6 字的「细分方向」标签（如"单机游戏""美食探店""硬核科技"）

**A2. 告知用户查询范围**

> 📊 关键词「**XXX**」搜索到 **N 个**B站账号（按粉丝数排序 | 第 {page} 页），已自动分为 **M 个**类别：

**A3. 按分类分组渲染 Markdown 表格（全部展示，⚠️ 严禁截断）**

每个分类单独一个表格，格式如下：

```markdown
### 🎮 游戏类（X 个）

| #   | 昵称                   | 粉丝数  | 等级 | 性别 | 直播间ID  | 简介                     | 是否UP主 | 细分方向   |
| --- | ---------------------- | ------- | ---- | ---- | --------- | ------------------------ | -------- | ---------- |
| 1   | [昵称](space_url)      | 105.2w  | Lv6  | 男   | 21894129  | 简介文字（超30字截断）   | 是       | 单机游戏   |
```

**序号规则：**
- 全局连续编号，不按分类重置
- 按分类顺序 + 分类内粉丝数降序连续编排

**分类输出顺序：** 按分类内账号数降序排列（账号数多的分类在前）

**数字格式化规则：**
- `< 10000`：原始数字（如 `320`）
- `≥ 10000`：`x.xw` 格式（如 `1.2w`）

**昵称规则：** `[昵称](space_url)`；昵称直接使用，不截断

**性别规则：** `gender=1` 显示「男」，`gender=2` 显示「女」，`gender=3` 显示「未知」

**是否UP主规则：** `is_uploader=true` 显示「是」，`is_uploader=false` 显示「否」

**直播间ID规则：** 直接显示原始数字；`live_room_id=0` 时显示 `-`

**简介规则：** 超过 30 字截断并加 `...`；无简介显示 `-`

**A4. 分类汇总表（紧接在最后一个分类表格之后）**

```markdown
## 📈 分类汇总

| 分类 | 账号数 | 总粉丝数 | 头部账号 |
|------|--------|---------|----------|
| 🎮 游戏 | 12 | 2821w | 渗透之C君 |
```

- 总粉丝数同样使用 `x.xw` 格式
- 头部账号为该分类内粉丝数最多的账号昵称

**A5. 翻页提示（条件输出）**

- 若 `has_next` 为 true（当前页返回数据 ≥ 20 条）：
> 📄 回复「查看更多」获取更多{关键词}账号~

- 若 `has_next` 为 false（数据已完全输出）：**不输出翻页提示，不提及「查看更多」**。

**⚠️ A1~A5 缺一不可，必须在同一轮输出中连续完成。若缺少任一步骤，视为执行错误，必须重新补充输出。**

### 情况 B：users 数量 = 0（无匹配结果）

**B1. 抱歉提示 + 拓词推荐**

> 😔 抱歉，未找到与「**XXX**」相关的B站账号，你可以尝试用更短或更宽泛的关键词重试（扩展词1,扩展词2,...扩展词10）

AI 必须生成**固定 10 个**扩展词，2~6 字，英文逗号分隔，不得少于 10 个。

## Step 4：记录页码（内部逻辑，不展示给用户）

将当前页码记录在上下文中。当用户回复「查看更多」时，页码 +1 再次调用脚本。
