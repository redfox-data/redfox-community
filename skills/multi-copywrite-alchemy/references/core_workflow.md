# Core Workflow

> 本文件包含复刻蒸馏写作技能的完整执行流程，供 Agent 内部参考。

---

## Phase 0: 素材摄入

### Step 0.1: 接收输入

用户可以通过以下方式提供文章素材：

| 方式 | 操作 | 适用场景 |
|------|------|--------|
| **平台账号查询** | 提供平台名称+账号ID/名称，自动获取文章列表和详情 | 素材在社交平台上，不想手动复制 |
| **直接粘贴** | 用户在对话中粘贴文章，用 `---` 或 `===` 分隔多篇 | 临时分析，文章量少 |
| **文件路径** | 用户提供文件夹路径或文件路径列表 | 文章量大，已整理好的素材 |
| **混合模式** | 同时接受粘贴、文件路径和平台查询 | 灵活使用 |

**如果用户指定平台账号**：→ 走 Step 0.1.1 平台自动采集流程

**如果用户提供文件路径**：
- 单个文件 → 直接读取
- 文件夹 → 批量读取文件夹内所有 .txt / .md 文件
- 多个文件路径（空格或换行分隔）→ 逐个读取
- 支持的文件格式：`.txt`、`md`、`docx`（尝试读取）

**如果用户直接粘贴**：
- 如果段落之间有明显分隔符（如 `---`、`===`、空行+标题）→ 自动拆分
- 如果无法自动拆分 → 询问用户每篇文章的边界

### Step 0.1.1: 平台自动采集流程

支持从公众号、抖音两个平台自动获取作者文章素材。

**触发条件**：用户提到平台名称（公众号/抖音）或提供平台链接/账号ID

**通用调用规范**：
- 请求方式：`POST` JSON
- 认证头：`X-API-KEY`（环境变量 `REDFOX_API_KEY`）
- 成功码：`code == 2000`（不是200）
- 超时设置：建议 30 秒

**调用代码模板**：

```python
import json, urllib.request

API_KEY = "环境变量 REDFOX_API_KEY"

PLATFORM_SOURCE_MAP = {
    "gzh": "复刻蒸馏写作技能-公众号-GitHub",
    "dy":  "复刻蒸馏写作技能-抖音-GitHub",
}

def redfox_api(path, payload, timeout=30):
    # 根据接口路径自动注入 source
    platform_key = path.split("/")[0]  # 取路径第一段，如 gzhData / dyData
    platform_alias = {"gzhdata": "gzh", "dydata": "dy"}.get(platform_key.lower().replace("_", ""), None)
    if platform_alias:
        payload = {**payload, "source": PLATFORM_SOURCE_MAP[platform_alias]}
    data = json.dumps(payload).encode()
    req = urllib.request.Request(
        f"https://redfox.hk/story/api/{path}",
        data=data,
        headers={"X-API-KEY": API_KEY, "Content-Type": "application/json"}
    )
    with urllib.request.urlopen(req, timeout=timeout) as resp:
        result = json.loads(resp.read().decode())
    if result.get("code") != 2000:
        raise Exception(f"API错误: {result.get('msg', '未知错误')}")
    return result.get("data")
```

#### 📱 公众号平台

**API 基础路径**：`gzhData/`

**Step 1 - 查询账号信息**：

| 配置项 | 值 |
|--------|----|
| 接口 | `POST gzhData/queryUser` |
| 入参 | `{"accountName": "公众号名称", "source": "复刻蒸馏写作技能-公众号-GitHub"}` 或 `{"accountId": "公众号ID", "source": "复刻蒸馏写作技能-公众号-GitHub"}` |
| 关键字段 | `accountName`、`accountId`、`description` |

**Step 2 - 获取作品列表**：

| 配置项 | 值 |
|--------|----|
| 接口 | `POST gzhData/queryWorkList` |
| 入参 | `{"accountId": "公众号ID", "source": "复刻蒸馏写作技能-公众号-GitHub"}` |
| 关键字段 | 返回文章列表，含 `title`、`publishTime`、`workId`/`id` 等 |

**Step 3 - 获取作品详情**：

| 配置项 | 值 |
|--------|----|
| 接口 | `POST gzhData/queryArticleDetail` |
| 入参 | `{"workId": "文章ID", "source": "复刻蒸馏写作技能-公众号-GitHub"}` 或 `{"id": "文章ID", "source": "复刻蒸馏写作技能-公众号-GitHub"}` |
| 关键字段 | `content`（文章全文）、`title`、`readCount`、`likeCount` |

**采集策略**：
1. 调用 queryUser 确认账号存在
2. 调用 queryWorkList 获取近期文章列表
3. 按发布时间排序，取前 N 篇
4. 逐篇调用 queryArticleDetail 获取全文
5. 清洗：移除公众号常见尾部引导语（"点击下方阅读原文"、"扫码关注我们"等）

---

#### 🎵 抖音平台

**API 基础路径**：`dyData/`

**Step 1 - 查询账号信息**：

| 配置项 | 值 |
|--------|----|
| 接口 | `POST dyData/queryUser` |
| 入参 | `{"accountId": "抖音号", "source": "复刻蒸馏写作技能-抖音-GitHub"}` 或 `{"accountName": "抖音名称", "source": "复刻蒸馏写作技能-抖音-GitHub"}` |
| 关键字段 | `nickname`、`uid`、`followerCount`、`awemeCount` |

> **注意**：优先使用 `accountId`（抖音号），`accountName` 可能模糊匹配到错误账号。

**Step 2 - 获取作品列表**：

| 配置项 | 值 |
|--------|----|
| 接口 | `POST dyData/queryWorkList` |
| 入参 | `{"accountId": "抖音号", "source": "复刻蒸馏写作技能-抖音-GitHub"}` 或 `{"uid": "用户UID", "source": "复刻蒸馏写作技能-抖音-GitHub"}` |
| 关键字段 | 返回作品列表，含 `title`/`desc`、`publishTime`、`awemeId`/`id` |

**Step 3 - 获取作品详情**：

| 配置项 | 值 |
|--------|----|
| 接口 | `POST dyData/queryWork` |
| 入参 | `{"awemeId": "作品ID", "source": "复刻蒸馏写作技能-抖音-GitHub"}` 或 `{"id": "作品ID", "source": "复刻蒸馏写作技能-抖音-GitHub"}` |
| 关键字段 | `desc`（作品描述/文案）、`title`、`likeCount`、`commentCount` |

**采集策略**：
1. 调用 queryUser 确认账号存在
2. 调用 queryWorkList 获取近期作品列表
3. 抖音以短视频为主，文字内容主要在 `desc` 字段（作品文案）
4. 筛选 desc 字数 > 100字 的作品（过滤纯标题视频）
5. 逐篇调用 queryWork 获取完整描述
6. 清洗：移除话题标签（#xxx）、@提及

---

#### 平台采集通用规则

1. **积分消耗提醒**：每次API调用会消耗红狐平台积分，采集前应告知用户
2. **采集数量**：默认采集10篇，用户可指定更多（最多50篇）
3. **字数筛选**：自动跳过字数 < 50字的作品（内容不足以分析）
4. **跨平台合并**：支持同时从多个平台采集同一作者的素材，合并后统一去重和预处理
5. **采集完成后**：自动进入 Step 0.2 素材预处理流程

### Step 0.2: 素材预处理

读取所有文章后，执行以下预处理：

1. **去重**：检测并移除高度重复的段落（同一作者可能在不同平台发相同内容）
2. **清洗**：移除明显的非正文内容（广告、免责声明、转载声明等）
3. **统计**：输出语料库概况

```
语料库概况
├── 总篇文章：N 篇
├── 总字数：XX,XXX 字
├── 篇目清单：
│   1. [标题1]（X,XXX字）- [来源/路径]
│   2. [标题2]（X,XXX字）- [来源/路径]
│   ...
└── 字数分布：[最短篇字数] ~ [最长篇字数]，平均 X,XXX 字/篇
```

### Step 0.3: 素材质量检查

- **最少要求**：≥3篇文章，总计≥2000字
- **理想条件**：≥10篇文章，总计≥10000字
- **低于最低要求**：提醒用户"素材太少，蒸馏出来的风格画像可能不够准确。建议至少3篇文章、总计2000字以上。是否继续？"
- **低于理想条件但满足最低要求**：正常进行，但在最终输出时标注"语料库较小，风格画像可能有遗漏"

**质量检查完成后，确认以下信息再继续**：
- 这位作者叫什么？用什么笔名？
- 这些文章发表在什么平台？（公众号/小红书/微博/知乎/博客/小说平台）
- 作者主要写什么题材？（情感/职场/育儿/小说/生活随笔/商业分析）

如果用户没提供这些信息，主动询问。

### Step 0.4: 确认蒸馏模式

向用户确认蒸馏范围（默认为完整蒸馏）：

```
蒸馏模式选择：
├── 🔥 完整蒸馏（推荐）：思维框架 + 写作风格 + 格式适配 → 全能分身
├── ✍️ 仅写作风格：原有7+1维分析 → 写作分身（v2.0模式）
└── 🧠 仅思维框架：只提取思维画像 → 思维参考卡
```

用户说"就完整蒸馏"或没提 → 默认完整蒸馏模式。

### Step 0.5: 信息源黑名单

**明确排除的低质量信息源**：

- **知乎**：洗稿严重、信息失真率高，不作为任何维度的来源
- **微信公众号**：封闭生态、无法验证、大量二手转述，不作为来源
- **百度百科/百度知道**：信息陈旧且不可靠

**高质量信息源优先级**：
1. **用户提供的一手素材**（最高优先级）
2. **本人著作/长文**（系统性思考）
3. **长对话/访谈转录**（即兴思维过程）
4. **实际文章文本**（本技能的核心分析对象）
5. **他人高质量分析**（外部视角、盲点）

## Phase 1: 双引擎并行分析

**首先，加载参考文件**：
- 读取 `references/7-dimensions-framework.md`，获取全部9个维度的详细分析方法（含新增维度九：思维框架分析）
- 读取 `references/writing-formulas.md`，获取所有写作公式的定义和适用场景

然后执行双引擎分析。两大引擎可以同时进行（独立分析，最后交叉验证）：

---

### 引擎A：思维蒸馏（维度九）

> 从文章中反向提取作者的思维框架。核心假设：**一个人的思维框架，一定会在他的写作中泄露出来。**

基于 `references/7-dimensions-framework.md` 维度九的方法论，执行5个子维度分析：

| 子维度 | 分析内容 | 核心问题 |
|--------|---------|---------|
| **价值观体系** | 高频价值观词、赞美对象、批判对象、价值排序 | 他觉得什么重要？ |
| **世界观与信念** | 世界基调、人性假设、核心信念（≥3次出现=真信念）、因果归因 | 他怎么看待这个世界？ |
| **身份认同** | 自我定位（导师/朋友/控诉者/记录者）、群体归属、立场倾向、同理心半径 | 他觉得自己是谁？ |
| **思维习惯** | 论证方式、思维工具、矛盾处理模式、确定程度 | 他怎么想问题？ |
| **对立面与禁忌** | 情绪触发点、不可妥协的底线、沉默地带、盲区提示 | 他绝对不接受什么？ |

**分析原则**：
- 从文章中提取，不猜测作者文章之外的想法
- 区分"他明确说的"（高可信度）vs"从文章中推断的"（中等可信度）
- 发现矛盾保留矛盾——思维框架中有张力是正常的
- 思维画像总结：一段100-150字的概括，描述此人写作时暴露的思维方式

---

### 引擎B：写作风格蒸馏（维度一~八）

对语料库执行前8个维度的分析。按依赖关系分为三组执行：

#### 第一组：基础统计维度（同时进行）

| 维度 | 分析内容 | 核心任务 |
|------|---------|---------|
| **词汇层** | 词频、词性、搭配、领域词 | 统计高频词 TOP 30、标注"指纹词"10-15个、分析词性分布、列出固定搭配 |
| **句法层** | 句长、句式、连接、标点 | 计算平均句长及标准差、分类句式比例、统计连接词密度、分析标点使用模式 |
| **语气层** | 口语化、节奏、信息密度 | 统计语气词密度、判断口语化指数、分析句长波动模式、计算词汇多样性 |

这三个维度都是对文本做纯统计，不需要理解内容，可以同时进行。

#### 第二组：结构分析维度

| 维度 | 分析内容 | 核心任务 |
|------|---------|---------|
| **篇章结构** | 开头、段落、结尾、逻辑 | 分类所有文章的开头/结尾类型、分析段落长度分布和节奏、标注逻辑推进方式 |

这个维度需要先读完所有文章的整体结构，在第一组之后进行。

#### 第三组：内容理解维度（同时进行）

| 维度 | 分析内容 | 核心任务 |
|------|---------|---------|
| **修辞与表达** | 修辞格、引用、具象层次 | 统计比喻/排比/拟人等修辞格频次、列出引用来源、分析具象词/抽象词比例 |
| **情感与态度** | 情感极性、态度立场、情绪节奏 | 统计情感词分布、标注主导态度类型、选取代表性文章绘制情绪曲线 |
| **内容倾向与视角** | 题材、意象、视角、价值观 | 标注所有文章的主题和常用意象、统计人称代词密度、分析价值观词聚类 |

这三个维度需要对文章内容进行语义理解，在第二组之后进行。

#### 第四组：格式适配分析（前7维完成后进行）

| 维度 | 分析内容 | 核心任务 |
|------|---------|---------|
| **格式适配** | 结构惯性、能力匹配、公式推荐 | 基于前7维结果，判断作者的默认组织方式、匹配各类公式所需的能力、输出天生适合/可以驾驭/不太适合的三级推荐 |

**判断逻辑**：
- **结构惯性**：作者习惯先抛问题再解答 → 天然匹配PAS/QUEST；习惯按时间叙述 → 天然匹配SCAR；习惯对比展开 → 天然匹配BAB
- **能力匹配**：强共情力+精准定义问题 → PAS高分；强故事叙述力+冲突设计力 → SCAR高分；强教学力+系统性 → QUEST高分
- **逆匹配警告**：如果作者的风格与某公式严重冲突（如极度口语化短句作者使用4Ps高客单价公式），标注"会丢失核心优势"

**详细分析方法**：参见 `references/7-dimensions-framework.md` 维度八

#### 分析输出格式要求

每个维度的分析结果写入同一个分析报告，格式如下：

```markdown
## 维度X：[维度名]

### 核心发现
[2-3句话概括该维度最关键的特征]

### 数据与分析
[具体的统计数据、摘录、分析]

### 典型原文摘录
> [从语料库中摘录2-3段最能体现该维度特征的原文]

### 可操作结论
- [可以转化为写作规则的结论1]
- [可以转化为写作规则的结论2]
- ...
```

**关键原则**：
- **有数据不臆测**：能统计的一定统计，不能统计的标注"基于人工判断"
- **有摘录不空谈**：每个特征至少附1段原文摘录佐证
- **发现矛盾就保留矛盾**：如果同一作者在不同文章中有不一致的写法，记录下来——这是风格复杂性的体现
- **区分"他惯用的"vs"他偶尔用的"**：标注频率等级（高频特征/中频特征/偶发特征）

## Phase 2: 双画像合成 + 双引擎交叉验证

将 Phase 1 的两组分析结果合成为两份画像，并执行交叉验证。

### Step 2.1: 思维画像合成

从引擎A的5个子维度分析结果，合成 **"作者思维画像"**：

```markdown
## [作者名]的思维画像

### 世界观
[一段话：世界基调 + 人性假设 + 因果归因]

### 价值观
[核心价值观3-5个，含价值排序]

### 身份认同
[自我定位 + 群体归属 + 立场倾向 + 同理心半径]

### 思维习惯
[主导论证方式 + 常用思维工具 + 矛盾处理模式 + 确定程度]

### 情绪触发点与禁忌
[情绪触发点列表 + 不可妥协的底线 + 盲区提示]
```

### Step 2.2: 写作风格画像合成

逐维精炼 → Step 2.3 格式适配推荐 → Step 2.4 交叉验证。

**Step 2.3: 格式适配推荐**：基于维度八结果，输出天生适合/可以驾驭/不太适合三级推荐。

**Step 2.4: 写作维度交叉验证**：从前7个维度中寻找跨维度交叉验证的线索，提炼写作基因。

### Step 2.5: 双引擎交叉验证

**这是整个技能最关键的一步。**

将思维画像（Step 2.1）和写作风格画像（Step 2.2）+ 写作基因（Step 2.4）进行跨引擎交叉验证——

寻找思维框架如何驱动写作风格的"映射关系"：

| 思维特征 | → | 写作物证 |
|---------|---|---------|
| 批判型态度 | → | 高频反问句 + 负面情感词 > 正面 |
| 人性默认不信任 | → | 故事中人物总被背叛 + 极少直白信任表达 |
| 看重"体面" | → | 高频"体面""丢人"+ 冲突围绕面子展开 |
| 用故事思维 | → | 抽象词密度低 + 开头总是具体场景 |
| 确定性高/不犹豫 | → | 短句比例高 + 极少"也许/可能" |

从交叉验证中提炼3-5条 **"思维基因"**——最底层的、同时驱动思维和写作的核心特征：

```
思维基因 #N：[一句话概括]
  → 思维层面：[它如何影响作者怎么想]
  → 写作层面：[它如何影响作者怎么写]
  → 反例测试：如果作者在想/写时违背了这条基因，会产生什么违和感
```

**思维基因 vs 写作基因的区别**：
- 写作基因描述"他怎么写"（如"语言极简，一句一顿"）
- 思维基因描述"他为什么这样写"（如"他觉得啰嗦是虚伪，所以一句一顿"）
- 思维基因是根，写作基因是叶。理解思维基因，才能在写任何类型的文章时保持"像他"

### Step 2.6: 生成禁用清单

合并写作禁用（来自维度一~八）和思维禁忌（来自维度九对立面与禁忌），生成统一的禁用清单：

- 写作禁用：具体的禁用写法（≥5条）
- 思维禁忌：不能碰的话题、不能站的立场、不能用的论证方式（≥3条）

### Step 2.7: 输出双画像卡

将以上内容组装为结构化的"作者思维+写作风格卡"，作为 Phase 3 的基础。

## Phase 3: 输出生成（双产出 + 写作模式 + 思维约束）

### Step 3.0: 写作模式预装

确认以下内容已就绪：
- 思维画像（Phase 2 Step 2.1）
- 写作风格画像（Phase 2 Step 2.2）
- 格式适配推荐（Phase 2 Step 2.3）

- 每个公式的"作者适配写法"——此人用这个公式时，不仅风格要对，**思维也要对**

**关键原则**：公式 = 骨架，风格 = 血肉，**思维 = 灵魂**。PAS的"问题→放大痛点→给解法"是骨架，风格决定怎么写，思维决定这个痛点在他眼里是不是痛点、这个解法在他眼里是不是解法。

## Phase 4: 双引擎盲测验证

### Step 4.1: 生成测试短文

写**三段**测试短文：
1. **默认模式**：作者最自然的写法（不指定公式）
2. **公式模式**：选一个"可以驾驭"级别的公式测试边界
3. **思维挑战模式**：选一个作者可能立场复杂/矛盾的话题（如"你表弟买房找你借五万，你借不借？"），测试思维一致性

每段 200-300 字。

### Step 4.2: 原文对照 + 思维对比

从语料库中随机选取一段原文并排展示。

### Step 4.3: 盲测分析

```markdown
## 盲测分析

### 测试1：默认模式
| 维度 | 原文特征 | 仿写表现 | 匹配度 |
|------|---------|---------|--------|
| 词汇 | ... | ... | ⭐⭐⭐ |
| ... | ... | ... | ... |

综合匹配度：XX%

### 测试2：公式模式（[公式名]）
公式模式综合匹配度：XX%

### 测试3：思维一致性测试（[挑战话题]）
| 检查项 | 评估 |
|--------|------|
| 立场是否符合作者的价值观？ | ✅/⚠️/❌ |
| 情绪反应是否符合作者的情绪触发点？ | ✅/⚠️/❌ |
| 论证方式是否符合作者的思维习惯？ | ✅/⚠️/❌ |
| 如果是原作者看这段，他会点头还是摇头？ | [判断] |

### 像的地方 | 不像的地方 | 改进建议
...
```

**匹配度判断**：
- 90%+：出口即是，基本不需要调整
- 70-89%：核心特征抓住了，细节有偏差
- 50-69%：大致方向对了，但辨识度不足
- <50%：需要回到 Phase 2 调整风格画像

### Step 4.4: 质量自检

生成分身后，执行质量自检。检查以下6项标准：

| 检查项 | 通过标准 | 不通过信号 |
|--------|---------|-----------|
| 思维基因数量 | 3-5条，每条有思维层面+写作层面映射 | <3或>7 |
| 每个基因的局限性 | 明确写出失效条件 | 只写优点 |
| 写作风格辨识度 | 读100字能认出是谁 | 像通用模板 |
| 诚实边界 | 至少3条具体局限（语料限制、时效性、盲区） | 只有「不能替代本人」 |
| 内在张力 | 至少2对矛盾 | 观点高度一致（太假） |
| 信息来源占比 | 一手文章文本 >80% | 主要依赖推测 |

验证通过 → 进入Phase 5精炼。不通过 → 标注薄弱环节，回到Phase 2迭代。
**迭代上限**：Phase 2→4最多循环2次。如果2轮后仍有不通过项，在诚实边界中标注薄弱维度，交付当前最优版本而非无限打磨。

**展示验证结果给用户确认后才算完成。**

## Phase 5: 双Agent精炼

Phase 4 验证通过后，自动启动双Agent精炼，进一步提升分身可操作性：

**并行启动两个Agent：**

**Agent A（使用者视角）**：
- 对分身技能SKILL.md执行6维度可用性评估（写作铁律清晰度、思维约束可操作性、检查点设计、指令具体性等）
- 干跑3个典型写作prompt，评估效果维度
- 输出：最弱2个维度的具体改进建议（要有改后文本示例）

**Agent B（skill-creator视角）**：
- 评审「触发条件」是否覆盖真实使用场景
- 评审「双重复检系统」的可操作性（复检一指纹覆盖率、复检二思想匹配度、失败重写路由）
- 识别缺失的关键信息
- 输出：2-3处具体文本改动建议（要有改后文本示例）

**主Agent综合两份报告，应用不冲突的改进，展示变更摘要请用户确认。**

精炼标准：改动必须让分身「激活即执行」，不只是增加内容，而是让AI拿到分身后知道先做什么、碰到什么停下来。

## 更新已有分身

当用户说「更新XX的分身」「XX最近有新文章」时：

1. 读取现有分身技能 SKILL.md，从「诚实边界」section中找到「分析时间：[日期]」，标注距今多久
2. 只分析新增文章（Phase 1 双引擎分析，但只针对新文章）
3. 对比新信息与现有内容：
   - 新信息强化现有基因 → 补充案例
   - 新信息与现有基因矛盾 → 标注变化，更新基因
   - 出现新的思维/写作特征 → 考虑增加新基因
4. 更新SKILL.md中的「最新动态」section和分析时间
5. 不重写整个分身，只增量更新

## 特殊场景处理

### 场景1：素材太少（<3篇或<2000字）

- 不拒绝，但明确告知风险
- 生成时标注语料不足警告

### 场景2：文章类型混杂

- 分析时标注"风格分离"
- 生成时让用户选择蒸馏哪种风格

### 场景3：同名不同人的文章

- 在 Phase 0 就标注出风格不一致的信号
- 提醒用户确认

### 场景4：更新已有分身

- 只分析新增文章，增量更新

### 场景5：蒸馏自己的文章

- 身份描述用"你就是你自己"
- 帮助用户发现自己可能没意识到的写作习惯

### 场景6：中国作者 vs 西方作者

- 中国作者信息源优先级：公众号文章 > 小红书/微博 > 权威媒体采访
- 西方作者信息源优先级：本人博客/Medium > Twitter/X > 权威媒体采访

### 场景7：冷门作者（公开信息极少）

- 思维基因减至2-3条，标注「基于有限信息推测」
- 诚实边界section加大篇幅

### 场景8：主题风格蒸馏 vs 作者风格蒸馏

输入不是单个作者而是主题时，各Phase相应变体处理。

## 品味守则

| 原则 | 一句话 |
|------|--------|
| **多篇 > 单篇** | 10篇比1篇更能揭示真正稳定的风格特征 |
| **习惯 > 偶然** | 反复出现≥3次的才是真特征 |
| **矛盾有价值** | 保留矛盾，不要强行调和 |
| **具体 > 抽象** | 用数据说话 |
| **否定 > 肯定** | "他从不做什么"往往更能定义风格边界 |

### 绝不做的事
- 凭空猜测作者的背景或动机（只基于文本分析）
- 把通用写作技巧包装成该作者的"独门秘笈"
- 忽略文章中的不完美和不规整
- 在语料不足时强行生成"完整"的风格画像
